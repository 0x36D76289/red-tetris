# Repository Guidelines

## Project Structure & Module Organization
- `client/`: React UI, game logic under `game/`, Redux state in `redux/`, shared UI in `components/`, styles in `styles/`; entry files `App.js` and `index.js`.
- `server/`: Express + Socket.io gateway (`server/index.js`) with gameplay orchestration in `server/game/`.
- `__tests__/`: Jest suites for client/server modules. Vite outputs built assets to `dist/` (served by the server in production).

## Build, Test, and Development Commands
- `npm install` boots the workspace (uses `package-lock.json`).
- `npm run dev:server` launches the API/socket server on `PORT` (default 3000).
- `npm run dev:client` starts Vite on `VITE_PORT` (default 5173) with `/socket.io` proxied to the backend.
- `npm run build` bundles the client to `dist/`; run before `npm start` for production serving.
- `npm start` serves built assets + sockets; `npm run preview` inspects the `dist/` build via Vite.
- `npm run lint` runs ESLint across `server/` and `client/`; `npm test` / `npm run test:watch` execute Jest with coverage reporting.

## Coding Style & Naming Conventions
- JavaScript/JSX (ES2021), 2-space indentation, semicolons kept; favor `const`/`let`.
- React components use PascalCase filenames/components; Redux actions/reducers in camelCase; hooks prefixed with `use`.
- ESLint extends `eslint:recommended` + `plugin:react/recommended`; React-in-JSX-scope and prop-types checks are disabled. Keep socket event names consistent with server handlers.

## Testing Guidelines
- Jest with jsdom; React Testing Library is available via `jest.setup.js`. Tests live in `__tests__/` and follow `*.test.js` or `*.spec.js`.
- Coverage thresholds: 70% statements/functions/lines and 50% branches (see `jest.config.js`); target equal or higher coverage for new code.
- Before PRs, run `npm test` (or `npm run test:watch` locally) and keep tests deterministic—avoid timers unless required by game flow.

## Commit & Pull Request Guidelines
- Recent history is terse (`wip`, `new`); prefer clear, imperative subject lines such as `Add garbage-line handling` or `Fix socket reconnect`.
- PRs should include a concise summary, linked issue/room ID if relevant, test evidence (`npm test`, `npm run lint`), and UI screenshots for client-facing changes. Note any env var additions or migration steps.

## Configuration & Environment
- Copy `.env.example` to `.env`; set `PORT`, `NODE_ENV`, optional `VITE_PORT`/`VITE_PREVIEW_PORT`, and `CLIENT_ORIGIN` when the client is hosted separately. Do not commit secrets.
- In development, the client proxies `/socket.io` to the backend; ensure `npm run dev:server` is running. In production, `server/index.js` serves the `dist/` build generated by `npm run build`.

## Cahier des charges (exigences)

1) Objet du projet

Réaliser un jeu Tetris jouable en ligne (solo et multijoueur), basé sur une architecture client/serveur, en Full-Stack JavaScript uniquement (client navigateur + serveur Node.js), avec communication temps réel via socket.io.

⸻

2) Objectifs techniques attendus

Le projet doit démontrer :
	•	l’application de principes de programmation fonctionnelle (exigée) ;
	•	la mise en œuvre d’une logique asynchrone client/serveur ;
	•	des patterns réactifs (UI + gameplay) ;
	•	une démarche de tests unitaires avec contraintes de couverture.

⸻

3) Périmètre fonctionnel

3.1 Modes de jeu
	•	Solo : une partie peut être jouée avec un seul joueur.  ￼
	•	Multijoueur : plusieurs joueurs dans une même partie (room) ; plusieurs parties concurrentes doivent être possibles côté serveur.  ￼

3.2 Condition de victoire
	•	Pas de scoring (obligatoire) : la partie se termine quand il ne reste qu’un joueur vivant ; ce joueur est le gagnant.  ￼

3.3 Persistance
	•	Aucune persistance de données requise (pas de DB nécessaire).  ￼

⸻

4) Spécifications de gameplay

4.1 Plateau
	•	Chaque joueur possède son propre terrain : 10 colonnes × 20 lignes.  ￼

4.2 Pièces (Tetriminos)
	•	Utiliser les formes originales de Tetriminos et les règles de rotation originales.  ￼

4.3 Gravité & verrouillage
	•	Les pièces tombent à vitesse constante.
	•	Quand une pièce touche la pile, elle devient immobile seulement à la frame suivante, permettant un ajustement de dernière seconde.  ￼

4.4 Contrôles
	•	Gauche/Droite : déplacement horizontal
	•	Haut : rotation
	•	Bas : soft drop
	•	Espace : hard drop  ￼

4.5 Synchronisation multijoueur (obligation forte)
	•	Tous les joueurs d’une même partie doivent recevoir la même séquence de pièces, avec les mêmes positions et coordonnées, même si elles arrivent à des instants différents.

4.6 “Spectrum” (vision adversaire)
	•	Chaque joueur voit :
	•	les noms des adversaires ;
	•	une vue spectrum : hauteur du bloc le plus haut de chaque colonne (10 valeurs) ;
	•	cette vue se met à jour en temps réel.

4.7 Lignes de pénalité (multijoueur)
	•	Quand un joueur efface n lignes, les adversaires reçoivent (n − 1) lignes de pénalité indestructibles ajoutées en bas de leur terrain.  ￼

⸻

5) Gestion des parties (rooms) & cycle de vie

5.1 Entrée dans une partie via URL (obligatoire)
	•	Les joueurs rejoignent une partie via l’URL :
http://<server_name_or_ip>:<port>/<room>/<player_name>  ￼

5.2 Routeur (obligatoire)
	•	Exigence à respecter telle quelle :
“Use appropriate routers such as BrowserRouter or MemoryRouter for URL handling”  ￼

5.3 Host (hôte)
	•	Le premier joueur qui rejoint une room devient host.
	•	Le host contrôle le start et le restart de la partie.
	•	Si le host quitte, un autre joueur restant devient host.

5.4 Règle de verrouillage d’entrée
	•	Une fois la partie démarrée : aucun nouveau joueur ne peut rejoindre jusqu’au round suivant.  ￼

⸻

6) Exigences techniques (contraintes non négociables)

6.1 Stack et langage
	•	Projet développé entièrement en JavaScript, dernière version disponible.  ￼

6.2 Architecture réseau
	•	Modèle client/serveur.
	•	Communication : HTTP + socket.io (événements bidirectionnels).
	•	Le serveur doit servir via HTTP :
	•	index.html
	•	bundle.js
	•	assets statiques  ￼
	•	Le serveur exécute une boucle asynchrone gérant les événements socket.io.  ￼

6.3 Client (Front) — contraintes du PDF + tes exigences
	•	Le client est une SPA : le navigateur charge index.html (référence vers bundle.js) et aucun autre fichier HTML n’est échangé ; rendu et logique côté client.  ￼
	•	Interdictions HTML/DOM :
	•	pas de <TABLE /> (layout via grid ou flexbox)
	•	pas de bibliothèques de manipulation DOM (ex. jQuery)
	•	pas de Canvas
	•	pas de SVG
	•	Programmation fonctionnelle côté client (PDF) :
	•	interdiction d’utiliser le mot-clé this (sauf sous-classes de Error)  ￼
	•	logique board/pièces en fonctions pures  ￼

✅ Exigences ajoutées (imposées par toi)
	•	React obligatoire (pas Vue). (Le PDF autorise un framework moderne et cite React/Vue ; ici on fixe React.)
	•	Redux obligatoire — Redux Toolkit uniquement (store, slices, selectors, thunks/async si nécessaire). (Le PDF recommande Redux ; ici on le rend obligatoire.)  ￼
	•	Lodash obligatoire (dépendance installée + réellement utilisée). (Le PDF le propose comme optionnel ; ici on le rend obligatoire.)
	•	Code front absolument fonctionnel :
	•	uniquement des composants fonctionnels + hooks (zéro class component)
	•	règles de jeu dans un “engine” pure functions (immutabilité, pas d’effets de bord)
	•	effets de bord confinés (socket, timers, IO) et orchestrés par le layer “app”.

6.4 Serveur (Back)
	•	Code serveur en approche orientée objet via prototypes.
	•	Classes minimum attendues : Player, Piece, Game.  ￼
	•	Responsabilités serveur attendues : gestion jeux/joueurs, distribution des pièces, mises à jour spectrum.

Pour concilier “serveur prototype-based” et “code fonctionnel” : la structure (Player/Game) peut rester prototype-based, mais la logique de règles (collision, lignes, pénalités, génération) doit être implémentée en fonctions pures réutilisables.  ￼

⸻

7) Protocole réseau (socket.io) — à spécifier clairement

Le PDF impose de définir clairement le protocole et le partage de responsabilités.  ￼

7.1 Principes
	•	Le serveur est la source de vérité pour :
	•	la composition des rooms, host, start/restart
	•	la séquence de pièces (cohérente pour tous)
	•	l’émission des spectrums
	•	l’application des pénalités

7.2 Événements minimaux recommandés (exemple)
	•	Room
	•	room:join (client→server) { room, playerName }
	•	room:state (server→client) { players, hostId, status }
	•	room:hostChanged (server→room) { hostId }
	•	Game
	•	game:start / game:restart (host→server)
	•	game:started (server→room) { pieceSeed/sequenceRef, startAt }
	•	game:ended (server→room) { winnerId }
	•	Gameplay
	•	input (client→server) { type, t }
	•	state (server→client) { board, activePiece, next, gameOver }
	•	spectrum:update (server→room) { playerId, spectrum[10] }
	•	penalty (server→targets) { count }

(Le format exact est libre, mais doit couvrir les obligations ci-dessus.)

10) Critères d’acceptation (checklist finale)

Fonctionnel
	•	Rejoindre une partie via http://<host>:<port>/<room>/<player_name> fonctionne.  ￼
	•	Le routeur React gère l’URL avec BrowserRouter ou MemoryRouter.  ￼
	•	Le 1er joueur est host ; host peut start/restart ; host est réassigné s’il quitte.
	•	Après start, aucun nouveau joueur ne rejoint avant le round suivant.  ￼
	•	Plateau 10×20 ; pièces/rotations originales ; gravité constante ; lock à la frame suivante.  ￼
	•	Même séquence de pièces (mêmes positions/coordonnées) pour tous les joueurs d’une room.
	•	Spectrum temps réel + noms adversaires visibles.
	•	Pénalités : clear n lignes ⇒ adversaires reçoivent n−1 lignes indestructibles.  ￼
	•	Pas de scoring ; dernier joueur vivant gagne.  ￼

Technique / Front
	•	Front React en SPA.
	•	Redux Toolkit obligatoire (store RTK + slices + selectors) utilisé comme état central.
	•	Lodash obligatoire installé et utilisé.
	•	Zéro Canvas / SVG / jQuery / <TABLE> ; layout grid/flexbox.
	•	Côté client : pas de this (sauf Error) ; logique de jeu en fonctions pures ; code “fonctionnel”.  ￼

Technique / Back
	•	Serveur Node.js + socket.io ; boucle asynchrone d’événements.  ￼
	•	Serveur prototype-based avec au minimum Player/Piece/Game.  ￼
	•	Le serveur sert index.html, bundle.js et assets en HTTP.  ￼

Tests / Sécurité
	•	Couverture ≥ 70% (statements/functions/lines) et ≥ 50% (branches).
	•	Aucun secret en repo ; .env gitignored.  ￼
